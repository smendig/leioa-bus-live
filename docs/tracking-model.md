# Tracking model

Leioa Bus Live does not receive vehicle coordinates. It polls integer arrival estimates for every
stop and infers a plausible vehicle state from that projection. A recent API response is not
necessarily a fresh prediction: the upstream ETA vector commonly remains unchanged across several
15-second polls.

## Evidence base

The current model was reviewed against 339,223 snapshots captured between 29 April and 11 July 2026. The dataset contains 10.8 million stop polls across 64 service days. A chronological holdout
(training on May, evaluation on June and July) showed:

| Duration model                   | Mean absolute error | Median absolute error | P90 absolute error |
| -------------------------------- | ------------------: | --------------------: | -----------------: |
| Previous stop-pair constants     |              46.4 s |                30.9 s |            109.5 s |
| May median by stop pair          |              24.4 s |                16.0 s |             52.6 s |
| May median by line and stop pair |              22.6 s |                15.6 s |             47.5 s |

The measurements describe when the prediction engine changes its nearest target stop. They are not
GPS travel times and must not be presented as physical-position accuracy.

## Source semantics

The public arrival endpoint behaves like a computed, cached projection:

- ETAs are whole minutes and are repeated across consecutive polls.
- A change can affect different future stops by different amounts, so it is not just a mechanical
  timetable countdown.
- `IdBus = 0` occasionally represents a service without a usable vehicle identity and is excluded
  from map tracking.
- Partial stop failures can make a future stop disappear from one snapshot.
- The same stop pair can have materially different timing on different lines.

The safest interpretation is a hybrid schedule/operations prediction generated by the upstream
fleet-management system. The public payload does not prove whether GPS, manual control events, or
another operational signal fed a particular prediction.

## Model invariants

1. Identity is `busId + lineRef`; zero or missing vehicle identifiers are not physical vehicles.
2. Segment priors are keyed by `lineRef + previousStopId + nextStopId`.
3. A prior carries sample size and a P10/P50/P90 distribution, not only a point estimate.
4. Visual progress is monotonic while the target segment is unchanged.
5. ETA changes are evidence, but never a direct fraction of segment completion.
6. Repeated ETA values reduce freshness and confidence; they do not alone prove that a prediction
   is obsolete.
7. Suppression requires prolonged, mutually reinforcing stale signals. Uncertain vehicles remain
   visible in a holding or ambiguous state.
8. Diagnostics expose prediction age, prior source, uncertainty, and suppression evidence.
9. A vehicle missing from one or two complete snapshots is retained at its last position with lower
   opacity. It is removed after a third consecutive absence or restored immediately if it returns.

## Polling and degraded data

- Active service is polled every 15 seconds.
- An empty but otherwise valid network is polled every 30 seconds.
- Likely out-of-service periods and hidden browser tabs are polled every 60 seconds.
- Incomplete snapshots never advance or remove vehicles. After three consecutive partial snapshots,
  the UI reports degraded source data while retaining the last complete state.
- Empty responses during a published service window are reported as missing live data, not as proof
  that the service is not scheduled. This avoids hiding an upstream outage behind “Sin servicio”.

## Validation

Changes to tracking should be evaluated chronologically rather than on the same period used to
derive priors. Useful metrics are:

- segment-transition duration error;
- false suppression rate;
- monotonic marker movement;
- time spent in ambiguous/holding states;
- robustness to missing stop responses;
- calibration of reported confidence against subsequent forward transitions.

Without an external GPS or manual observation reference, replay can validate temporal consistency
but cannot establish exact map-coordinate accuracy.

Run the chronological validator against a local sharded capture with:

```bash
npm run tracking:validate -- captures/sessions/<session-id> \
  --cutoff=2026-06-01T00:00:00.000Z
```

Add `--emit-priors` to include the training-only P10/P50/P90 table in the JSON output. The cutoff is
exclusive for training: data before it trains the candidate model and data at or after it remains
unseen evaluation data.
